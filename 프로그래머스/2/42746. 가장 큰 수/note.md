## 1. 🔍 어떻게 시도했는가? (How)

### ❌ [FAIL / SO-SO] 최초 접근 로직
* **🤔 생각한 흐름:**
    * 숫자로 비교한다면 자릿수 계산을 해야 하는데, 차라리 String concatenation을 사용하는 게 더 효율적일 것이다.
    * Stream을 사용하자.
        * int[]를 IntStream으로 전환하고 이걸 다시 Stream<String>으로 전환하고
        * Comparator 익명 객체를 정의해서 비교 로직을 구현한 다음,
        * StringBuilder에 요소 단위로 append해서 단일 문자열로 반환하자!
    * 문자열 기준으로 단순 비교를 한다면 30과 3과 같은 경우를 제대로 정렬 못 한다.
        * 330 vs. 303으로 비교하도록 만들어야 한다.

* **⚠️ 왜 비효율적/실패했는가? (Self-Feedback):**
    * forEach(sb::append)보다 collect(Collectors.joining())이 더 효율적이다.
    * 1회성 문자열 이어붙이기는 + 연산자를 사용하는 게 좋다. 동일하게 효율적이고 가독성이 우수하다.
    * 익명 객체를 구현하기보다는 람다식을 활용하는 게 더 가독성이 좋다.
    * Stream은 오버헤드 코스트가 있기 때문에 더 효율적으로 처리하기 위해서는 반복문과 StringBuilder를 직접 이용하는 것이 좋다.
        * 반복문으로 int[]를 String[]로 전환하고
        * Arrays.sort에서 람다식을 사용하고
        * Edge case를 early exit하고
        * StringBuilder로 한 번에 최종 문자열을 만들고

### 🛠️ 내 코드에서 놓친 디테일 (Syntax / Edge Case)
* **🚨 엣지 케이스:**
    * [0, 0, 0]일 경우 "000"이 아니라 "0"을 반환해야 한다.

* **💡 문법/라이브러리 피드백:**
    * Modern Java Compiler는 String concatenation이 1회성일 경우, `+ 연산자`를 이용하는 것이 효율적이고 가독성이 좋다. `StringBuilder`를 사용해야 하는 경우는 Loop 밖에서 최초 한 번 `StringBuilder` 객체를 생성하는 경우이다.
    * 최종 문자열을 합칠 때 `forEach(sb::append)`보다 `collect(Collectors.joining())`이 더 효율적이다. 내부적으로 스트림의 크기를 고려해서 최적화된 StringBuilder를 사용하기 때문이다.

## 2. 🎯 한 줄 본질 (What)
* 이 문제는 결국 두 수 간의 조합을 통해 두 수를 정렬해야 때 숫자가 아닌 `문자열 이어붙이기`를 활용해서 `커스텀 정렬`을 구현하는 문제다.
