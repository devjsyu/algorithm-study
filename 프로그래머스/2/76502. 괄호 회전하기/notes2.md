## 1. 🔍 어떻게 시도했는가? (How)

### ❌ [FAIL / SO-SO] 최초 접근 로직
* **🤔 생각한 흐름:**
    * 회전이니까 기존 문자열을 2배로 늘려서 일부만 순회하는 방식을 사용할 수 있다.
    * 문자열 길이가 1,000 이하이니까 이중 반복문 구조로도 충분히 풀 수 있다.
    * 올바른 괄호 문자열인지 여부를 판별하는 메서드를 만든다?
    * 올바른 괄호 문자열인지 어떻게 판별? 여는 괄호를 스택으로 쌓아올리고, 닫는 괄호가 나올 때 스택에서 빼내는 과정 반복해서 스택이 최종적으로 비어있는지 여부로 판별한다.
    * char[]로 전환해야 할까? String.indexAt을 활용해도 충분할까?

* **Pseudo code**
```java
// 주어진 문자열 2배로 늘리기
// 1차 반복문 : 인덱스 i가 0부터 인덱스 s.length - 1까지 순회하기
// 2차 반복문 : 인덱스 j가 i부터 j + s.length - 1까지 순회하기
// private boolean isValid 메서드
    // 문자 배열 속 문자 원소에 대해 아래 과정 반복 수행
    // 여는 괄호를 스택으로 쌓아올리고, 닫는 괄호가 나올 때 스택에서 빼내기
    // 순회 종료 후 스택이 비어있는지 여부를 반환
```

* **⚠️ 왜 비효율적/실패했는가? (Self-Feedback):**
    * `solution` 메서드의 지역변수를 `isValid` 메서드에 파라미터로 주지 않으면 몰라. `Solution` 클래스의 멤버 변수가 아닌 이상 `isValid` 메서드에서 효율적으로 변수를 활용하지 못할 것 같아. 그래서 `isValid`를 사용하지 않고 이중 반복문에 모든 로직을 다 집어넣었어.

### ✅ [SUCCESS] 정답/모범답안 로직
* **✨ 수정된 흐름:**
    * 주어진 문자열 2배로 늘리고 char[] 문자 배열 만들기
    * 이중 반복문 만들기
    * private boolean isValid(char[] arr, int start, int end) 메서드 만들기
    * HashMap은 변치 않고 자주 참조되니 멤버 변수로 만들기
    * Stack 객체 생성 비용은 미미하니 isValid 메서드 내에 생성해도 무방
    * char[]와 Stack을 멤버 변수로 선언해놓고 solution 메서드에서 초기화해도 isValid 메서드에서 참조 가능. 이 경우 isValid에 필요한 메서드 파라미터 개수를 줄일 수 있다.

### 🛠️ 내 코드에서 놓친 디테일 (Syntax / Edge Case)
* **🚨 엣지 케이스:**
    * 홀수인 경우 계산 생략하고 바로 결론 낼 수 있다.
* **💡 문법/라이브러리 피드백:**
    * 현대 JVM은 Young Generation의 경우, 빠르게 GC가 처리할 수 있으니 lifespan이 짧은 객체를 부담없이 생성해도 괜찮다. 가독성을 우선시 해도 된다.
    * 레거시 Stack보다는 Deque를 사용하자.
    * Stack은 모든 메서드에 synchronized가 걸려 있는데, 정작 코딩테스트 환경에서는 싱글스레드이기 때문에 불필요한 오버헤드 발생한다.
    * 이중 반복문 속 내부 반복문을 탈출하기 위해서 `outerLoop:` `continue outerLoop;`을 사용할 수 있다.
    * isValid 메서드를 호출할 때마다 파라미터로 매번 똑같이 char[] arr를 넘겨주는 것은 의외로 오버헤드 코스트가 적다. 참조 변수의 주소값을 pass-by-value로 넘겨주기 때문에 고작 주소값 4byte만이 쓰일 뿐이다.isValid 메서드를 호출할 때마다 파라미터로 매번 똑같이 char[] arr를 넘겨주는 것은 의외로 오버헤드 코스트가 적다. 주소값을 pass-by-value로 넘겨주기 때문에 고작 주소값 4byte 또는 8byte만이 쓰일 뿐이다.
    * 지역 변수는 스택 영역에 임시로 저장되고, 메서드가 종료 되면 사라진다. 반면, 멤버 변수는 객체와 마찬가지 힙 영역에 저장되고 GC에 의해서 사라진다.
