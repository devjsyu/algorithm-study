## ❌ [FAIL / SO-SO] 최초 접근 로직
### 🤔 생각한 흐름:
* 문자 변경
    * Character 클래스의 메서드 활용
        * isAlphabet
        * toLowercase
        * trim
* 문자 `Palindrome` 여부 판별
    * 워낙 익숙해서 `Stack`을 사용하면 되겠다 싶은데, 이건 `Two Pointers`를 쓰면 비효율적인 걸까?
    * 왜 `Stack`을 쓰면 좋을까?
        * 가장 최근에 삽입한 원소부터 순서대로 접근할 수 있다.
        * 문제는 문자열이 대칭인지를 묻고 있다. 시작 방향과 끝 방향으로 각각 순회할 때 문자가 서로 일치해야 한다.
    * 서로 반대 방향으로 움직이는 `Two Pointers`로도 충분히 풀 수 있을 것이다.

### ⚠️ 왜 비효율적/실패했는가?:
* 별도 자료구조 없이 공간복잡도 O(1)로 구현할 수 있었다.
* 주어진 문자열을 Stream, List, array로 바꿀 필요 없이 charAt 메서드만으로 처리할 수 있었다.
* Primitive type `char` 대신 Wrapper class `Character`를 배열로 다루면 Autoboxing 비용이 불필요하게 생긴다.
* `Two Pointers`가 서로를 향해 이동하기 때문에 `p1 < p2`와 같이 배열의 절반만 검사해도 충분하다.

## ✅ [SUCCESS] 정답/모범답안 로직
### ✨ 수정된 흐름:
* 별도 자료구조 없이 주어진 문자열에 대해 인덱스 역할을 하는 left, right 포인터 선언하기.
* while loop을 통해 유효하지 않은 경우 넘어가기.
    * double while loop 구조이지만, 순회가 중첩되지 않고 한 방향으로 이동하기 때문에 O(N) 시간복잡도를 갖는다.
* 중간에 만날 때까지만(left < right) 주어진 문자열 길이의 절반만 비교 후 종료

## 🛠️ 내 코드에서 놓친 디테일
### 💡 문법/라이브러리 피드백:
* Autoboxing 비용을 고려해야 한다.
* List, Stream, array로 변환하지 않고 charAt만으로 처리할 수 있었다.

* 완벽히 정제된 리스트를 만들고 유효 문자열인지 판별하는 대신, 즉석에서 가공/필터링 하는 방식이 더 효율적이다.
