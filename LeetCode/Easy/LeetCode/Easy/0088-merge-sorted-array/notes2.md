## 1. 🔍 어떻게 시도했는가? (How)

### ❌ [FAIL / SO-SO] 최초 접근 로직
* **🤔 생각한 흐름:**
    1. `Two pointers`가 떠올랐다. 각 배열에 대해서 원소를 가르키는 포인터가 2개 있고, 각 포인터가 가르키는 원소를 비교해서 포인터를 이동하는 방식으로 풀면 어떨까?
    2. 두 개의 포인터를 어떻게 관리하지? 변수 2개를 일일이 다루는 걸까? 반복문을 어떻게 하지? m + n을 만족할 때까지? 각 포인터별 개수를 모두 소진할 때까지? 이 경우, 반복문이 여러 개가 될 텐데?
    3. `while (각 포인터의 한계까지 도달하기 전까지)` 비교해서 큰 쪽을 ++ 연산
    4. 관리해야 하는 변수와 반복문 조건이 2개 이상이라서 비효율적으로 느껴진다.
* **⚠️ 왜 비효율적/실패했는가? (Self-Feedback):**
    * **오답**
        * 배열 원소 복사가 아니라 배열 주소 복사를 했다가 뒤늦게 잘못 깨달음. Solution 클래스의 merge 메서드 scope로만 보면 문제 없어보이는데, 사실 채점 과정에서 merge 메서드는 스택 메모리에서 사라질 예정. 채점하는 클래스(ex. `Main`)의 메서드 스택에는 기존 nums1 변수가 그대로이다.
        * 인덱스 `edge case`에 오류가 있었다는 걸 디버거로 직접 해보고 나서야 깨달음. 직접 손으로 흐름을 쓰지 않고 머릿속으로 하려다 보니까 오히려 시간도 더 오래 걸리고 부정확했다.
    * **비효율**
        * while 반복문 안에 수많은 if 조건식이 나열되어 있어서 가독성도 안 좋고 실수할 여지도 많았다. 반면, 전형적인 merge sort 구현 방식은 반복문을 3개로 분리해서 깔끔하고 직관적이다. 굳이 단일 반복문 안에 모든 경우의 수를 다루기 위해 조건식을 복잡하게 구성할 필요 없이, 각 범위의 반복문을 나열해서 어차피 흐름에 따라 모든 경우의 수를 포괄할 수 있었다.
        * 후위연산자 ++를 제대로 쓰지 않았다. 코드가 불필요하게 길어졌다.
        * 메서드 초반 early exit을 위한 조건식이 이후 나오는 조건식과 중복되었다.

### ✅ [SUCCESS] 정답/모범답안 로직
* **✨ 수정된 흐름:**
    1. 이미 정렬된 배열에 대해서 값 비교가 있다면 `Two pointer`를 통해 중첩 반복문 없이 문제를 풀 수 있다.
    2. 변수 i, j, k를 관리하는 건 동일하나 반복문과 조건식을 더 간결하고 직관적으로 바꾼다.
    3. 서로 두 포인터를 비교할 수 있을 때, 하나의 포인터만 남을 때로 양분하여 반복문을 작성한다.

### 🛠️ 내 코드에서 놓친 디테일 (Syntax / Edge Case)
* **🚨 엣지 케이스:**
    * 반드시 직접 손으로 예시 문제풀이를 작성하자.
* **💡 문법/라이브러리 피드백:**
    * Java's pass-by-value : 메서드 내부에서 새로 주소 값을 할당하더라도, 메서드가 종료되면 해당 메서드를 호출한 영역의 메모리는 해당 주소 값이 할당되지 않는다.
