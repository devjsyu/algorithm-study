# 📝 [LeetCode 14] Longest Common Prefix 리뷰 노트

## 1. 어떻게 시도했는가? (How)

### ❌ [FAIL] 최초 접근 로직 (수평 탐색 기반)

* **생각한 흐름:**
  1. 배열을 전체 순회하며 **가장 길이가 짧은 문자열**을 찾는다.
  2. 그 문자열을 기준으로 다른 문자열들과 비교하며 범위를 점진적으로 줄여나간다.


* **왜 비효율적인가? (Self-Feedback):**
  1. **불필요한 선행 연산:** 세로 탐색을 하면 검사 도중 알아서 조기 종료되므로, 미리 가장 짧은 문자열을 찾느라 배열을 한 바퀴 돌 필요가 없다. (첫 글자부터 다르면 대형 낭비 발생)
  2. **복잡한 상태 관리:** 최소값, 인덱스 변수 등을 일일이 관리해야 해서 코드가 지저분해지고 버그 확률이 높아진다.



### ⭕ [SUCCESS] 모범답안 참고 후 정정 로직 (세로 탐색 기반)

* **수정된 흐름:**
  1. 임의로 배열의 첫 번째 문자열(strs[0])을 기준으로 삼는다.
  2. 기준 문자열의 글자를 하나씩 세로 방향으로 다른 문자열들과 비교한다.
  3. 다른 문자열의 길이를 벗어나거나, 하나라도 문자가 불일치하면 그 즉시 조기 종료(Early Return)한다.



### ⚠️ 내 코드에서 부족했던 디테일

* **시간 복잡도 오판:** 
  * 단순 중첩 루프라고 $O(N^2)$이라 생각했으나, 배열의 크기($N$)와 문자열의 길이($M$)가 서로 다르므로 정확히는 $O(N \times M)$이다.
* **엣지 케이스(Edge Case) 누락:**
  * 빈 배열([])이나 null이 들어왔을 때 프로그램이 터지는 방어 코드가 없었다.
* **API 내장 기능 오해:**
  * substring(0, i)는 i가 0일 때 예외 처리를 안 해줘도 자동으로 빈 문자열("")을 반환한다.

## 2. 한 줄 본질 (What)

> 이 문제는 결국 **세로 탐색(Vertical Scanning)** 구조에 **조기 종료(Early Return)** 조건을 심어 연산을 최소화하는 문제다.

## 3. 아차! 포인트 (Why)

* 모든 데이터를 완벽하게 정렬하거나 기준을 잡고 시작하려다 보니 오히려 선행 연산 비용(Overhead)이 커진다는 점을 간과했다.
* 이중 반복문이더라도 제어 변수가 다르면 제곱($N^2$)이 아니라 두 변수의 곱($N \times M$)으로 표현해야 함을 배웠다.

## 4. 다음에 보면? (Trigger)

* 문자열 배열에서 공통 분모를 찾을 때는 일단 **첫 번째 원소를 임의의 기준**으로 잡고 시작하자.
* 코딩 테스트 문제를 볼 때 N의 범위(데이터 개수)를 먼저 확인하는 습관을 들이자. $N \le 5,000$ 수준이라면 조금 투박한 이중 루프 구조라도 과감하게 설계를 시작하자.
