```
// 가장 길이가 작은 문자열을 찾는다
    // 문자열 배열을 순회하면서 각 문자열에 대해 length 메서드로 문자열 길이를 구한다
    // 순회하는 과정에서 비교를 통해 최소값 변수와 인덱스 변수를 갱신한다
    // 문자열 길이가 가장 작은 원소의 인덱스를 알아낸다
// charAt 메서드를 통해 Character 일치 여부 비교한다
    // 만약 최소값 원소의 인덱스에 도달하면 순회를 건너뛴다
    // 불일치 나오는 인덱스를 갱신한다
    // 다음 문자열 순회할 때 불일치 나오는 인덱스 직전까지 범위를 좁힌다
    // 모든 문자열을 순회하거나 불일치 나오는 인덱스가 0보다 작다면 순회를 종료한다
// 순회를 마치고 남은 인덱스 정보로 문자열의 일부를 출력한다
    // 남은 인덱스가 0보다 작다면 ""을 출력한다
```

위 풀이 로직이 왜 비효율적인가?
```
// 위 로직이 비효율적인 이유
    // Vertical Scanning을 한다면, 조기 종료가 가능하기 때문에, 가장 길이가 작은 문자열을 굳이 찾을 필요가 없다.
    // 또한, 최소값 변수와 인덱스 변수를 일일이 관리하고 업데이트 하는 방식은 코드를 불필요하게 복잡하고 지저분하게 만든다.
```

모범답안 풀이를 보고난 후 정리한 로직
```
// O(N^2) 시간복잡도이지만, 전체 개수가 제한적이기 때문에 충분하다.
// 배열 속 모든 문자열의 문자 요소를 확인하기
    // 배열 첫번째 문자열을 임의 기준으로 삼는다
    // 배열 첫번째 문자열의 문자 요소를 순회한다
    // 배열 속 모든 문자열에 각각 주어진 인덱스로 임의접근하여 문자를 비교한다
    // 주어진 인덱스가 문자열의 길이보다 큰 경우, 또는 하나라도 기준이 되는 문자와 불일치할 경우 순회를 종료한다
// 순회를 마치고 남은 인덱스 정보로 문자열의 일부를 substring으로 출력한다
    // 조건을 모두 만족하지 못하면 ""을 반환해서 조기 종료한다
    // 순회가 모두 통과하면 기준이 되는 배열 첫번째 문자열 전체를 반환한다
```
위 로직에서 부족한 점
```
// 시간복잡도는 O(N*M)이다. 비록 이중 반복문이지만 반복문의 변수가 서로 다르기 때문이다.
// 엣지케이스에 대한 고려가 부족하다. 빈 배열 입력에 대한 방어 코드가 누락되었다.
// substring 메서드에 대해 제대로 몰랐다. str.substring(0, 0)은 자동으로 빈 문자열("")을 반환한다.
```

## 2. 한 줄 본질 (What)
- 이 문제는 결국 `Vertical Scanning`으로 `조기 종료` 로직을 포함한 반복문을 구하는 거다.

## 3. 아차! 포인트 (Why)
- 내가 막혔던 이유는 `주어진 N의 크기를 간과하고, 시간복잡도를 오판`했기 때문이다. N의 크기를 보고 O(N^2)이라도 풀이 가능하다는 점을 놓쳤다.
- 또한, 이중 반복문이라도 반복문의 변수가 서로 다르다면 O(N^2)이 아니라 O(N*M)이라는 점이다. 

## 4. 다음에 보면? (Trigger)
- 이중 반복문이더라도 N의 크기가 2,000~10,000 사이라면 다음엔 빠르게 이중 반복문 풀이 로직이라도 풀이를 시작하자.
